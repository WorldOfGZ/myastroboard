# MyAstroBoard - Internationalization (i18n) System

## Overview

MyAstroBoard implements a structured multilingual system supporting English, French, Spanish, and German.
The i18n system ensures user-facing content can be translated while preventing hardcoded strings.

**Current Status**: ✅ Active implementation for weather alerts, astro-weather dynamic content, and events API translations.

## Architecture

### Components

```text
MyAstroBoard i18n System
│
├── Frontend (JavaScript)
│   ├── static/js/i18n.js         # Global i18n manager
│   ├── static/i18n/en.json       # English translations
│   ├── static/i18n/fr.json       # French translations
│   ├── static/i18n/es.json       # Spanish translations
│   └── static/i18n/de.json       # German translations
│
└── Backend (Python)
    └── backend/i18n_utils.py     # Translation utilities
```

### Language Support

| Language | Code | Status | Translation | Type |
|----------|------|--------|-------------|------|
| English | `en` | ✅ Active | Human | Default |
| French | `fr` | ✅ Active | Human | - |
| Spanish | `es` | ✅ Active | Google* | - |
| German | `de` | ✅ Active | Google* | - |

* using `scripts/translate_i18n_values.py`

## Frontend Implementation

### i18n.js - The Global Manager

The file `static/js/i18n.js`:

- Loads translation files from `static/i18n/[language].json`
- Detects browser language automatically
- Manages language switching with persistent storage (localStorage)
- Handles placeholders in translation strings
- Provides fallback to English for missing keys
- Dispatches events when language changes

### Using Translations in HTML

```html
<h2 data-i18n="astro_weather.section_title">Current Conditions</h2>
<p data-i18n="common.loading">Loading...</p>
```

### Using Translations in JavaScript

```javascript
const msg = i18n.t('common.loading');
const alert = i18n.t('weather_alerts.critical_dew_risk', { time: '14:30' });
const lang = i18n.getCurrentLanguage();
await i18n.setLanguage('fr');
```

## Backend Implementation

### i18n_utils.py

The module `backend/i18n_utils.py` provides:

- Translation loading from JSON files
- Dot-notation key lookup
- Placeholder replacement
- Language management per request
- Alert creation helpers with translated content

Example:

```python
from i18n_utils import I18nManager, get_translated_message

msg = get_translated_message('common.loading', language='fr')
manager = I18nManager('en')
caption = manager.t('weather_alerts.critical_dew_risk', time='14:30')
```

## API Language Contract

For endpoints returning translated user-facing content:

- Frontend sends current language via `?lang=<code>`
- Backend may fallback to `Accept-Language`, then default to `en`
- Backend resolves placeholders before serialization
- Responses should not return hardcoded English where i18n keys exist

Implemented examples:

- `/api/weather/alerts?lang=fr`
- `/api/weather/astro-analysis?hours=24&lang=fr`
- `/api/events/upcoming?lang=fr`

## Translation Keys Structure

Keys use dot notation:

```text
namespace.section.key
```

Example:

```json
{
  "common": {
    "loading": "Loading...",
    "error": "Error"
  },
  "astro_weather": {
    "section_title": "Current Conditions",
    "loading_message": "Loading...",
    "alerts_title": "Weather Alerts"
  }
}
```
